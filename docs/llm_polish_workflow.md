## 1. 现在的状态

当前副本目录：`/home/ub/cx_ws/notes/cleaned`

这一版已经完成的确定性清洗：

- 删除语雀锚点标签，如 `<a name="..."></a>`
- 删除 `<br>` / `<br />`
- 清理常见图片链接后面的 `#...` 长参数

- 不改原始目录，只改副本目录

当前剩下的主要问题，已经不是“导出残留”，而是“阅读体验”：

- 某些段落可以拆得更顺
- 某些表格可以更像正常 Markdown
- 某些列表、标题层级可以更稳定
- 个别文件仍然有“像课堂笔记”的跳跃表达，不够适合长期阅读

这类问题更适合交给 LLM，而不是继续扩脚本。

## 2. 什么时候用 LLM，什么时候不用

适合 LLM 做的：

- 只优化可读性
- 只优化 Markdown 排版
- 拆分长段落
- 规范标题、列表、表格说明文字
- 保持技术内容不变的前提下，让文档更像人工整理稿

不适合直接让 LLM 放开改的：

- 改代码逻辑
- 改命令参数
- 改图片链接
- 改专业术语
- 改知识结论
- 擅自删减“看起来重复但其实有用”的技术内容

结论：

- “清洗导出痕迹”用脚本更稳
- “整理成适合人读的 Markdown”用 LLM 更合适
- 最稳的方案是：脚本先做确定性清洗，LLM 再逐文件润色

## 3. 不可违反的约束

给 LLM 时，必须明确要求：

- 不改知识内容
- 不改代码含义
- 不改图片链接
- 不删除代码块
- 不补充你自己没有写过的知识点
- 不擅自总结压缩内容
- 不把技术笔记改写成“教程风格长文”
- 只输出最终 Markdown，不要解释

## 4. 单文件提示词模板

适用于中小文件，推荐直接逐文件处理。

```text
你现在要做的是“技术 Markdown 笔记清洗”，不是改写知识内容。

目标：
把下面这份 Markdown 笔记整理成更适合人类阅读的版本，去掉导出感和课堂速记感，让它更像一份正常的 Markdown 学习笔记。

严格约束：
1. 不改变知识含义。
2. 不删除有效内容。
3. 不新增原文中没有的知识点。
4. 不修改代码逻辑、命令、接口名、术语、链接、图片链接。
5. 不删除图片。
6. 不把代码块改成普通文本。
7. 只优化排版、段落、标题、列表、表格、空行、语句衔接。
8. 遇到明显是作者原意但表达跳跃的地方，只做轻度整理，不要重写成另一篇文章。
9. 输出必须仍然是 Markdown。
10. 只输出整理后的 Markdown，不要加解释，不要加前言，不要加总结。

具体处理要求：
- 标题层级尽量稳定。
- 长段落适当拆分。
- 列表格式统一。
- 表格如果还能保留表格形式，就保留；如果表格已不适合阅读，可以改成等价的 Markdown 列表，但不要丢信息。
- 删除“导出痕迹感”的表达方式，但不要删技术内容。
- 保持代码块、图片、引用、链接位置基本稳定。

下面是原文 Markdown：

{{FILE_CONTENT}}
```

## 5. 超长文件分块提示词

适用于大文件，尤其是超过 `1200-1500` 行的文件。

建议分块时按“自然边界”切，不要硬切在代码块中间：

- 按一级标题或二级标题切
- 按章节切
- 保证每块内部代码块完整

分块提示词：

```text
你现在要整理一份超长技术 Markdown 笔记的其中一部分。

任务目标：
只清理这一段 Markdown 的阅读体验，让它更适合人类阅读。

严格约束：
1. 不改变知识含义。
2. 不删除有效内容。
3. 不新增原文没有的知识点。
4. 不修改代码逻辑、命令、术语、图片链接、普通链接。
5. 不删除图片和代码块。
6. 只做排版、列表、标题、段落、表格层面的整理。
7. 输出仍然必须是 Markdown。
8. 只输出整理后的 Markdown 片段，不要解释。
9. 必须保持这一段的开头和结尾语义连续，不能因为分块而乱改上下文。

下面是该文件的一个分块：

{{CHUNK_CONTENT}}
```

## 9. 建议工作流

推荐按下面顺序执行：

1. 选一个文件。
2. 把 `notes/cleaned` 中的文件内容喂给 LLM，不要喂 `notes/source`。
3. 使用上面的“单文件提示词模板”或“超长文件分块提示词”。
4. 只接收 Markdown 输出。
5. 对比前后差异，重点检查：
   - 代码块有没有变
   - 图片链接有没有变
   - 标题层级有没有乱
   - 表格信息有没有丢
   - 命令有没有被改写
6. 验证通过后，再覆盖副本文件。

## 10. 最重要的一条

如果目标是“适合复习、适合长期阅读”，LLM 是合适的。

如果目标是“完全可重复、零语义漂移”，脚本才是核心。

这批笔记最合理的路线不是二选一，而是：

`脚本做确定性清洗 + LLM 做逐文件阅读优化`
