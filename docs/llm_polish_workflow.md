# LLM Polish Workflow

这个文档说明：在完成脚本清洗之后，如何用 LLM 继续提升 `notes/cleaned/` 的可读性。

## 适用范围

适合交给 LLM 的工作：

- 优化 Markdown 排版。
- 拆分过长段落。
- 统一标题、列表、表格表达。
- 在不改知识内容的前提下，让笔记更适合复习和长期阅读。

不适合交给 LLM 的工作：

- 改知识结论。
- 改代码逻辑、命令参数、接口名、术语。
- 改链接、图片链接、图片含义。
- 删除看起来重复但实际有用的技术内容。

结论：

- 导出残留和格式噪音，优先用脚本处理。
- 阅读体验和版式整理，再交给 LLM 逐文件润色。

## 不可违反的约束

给 LLM 时，必须明确要求：

- 不改知识含义。
- 不删有效内容。
- 不增补原文没有的知识点。
- 不改代码块、命令、术语、链接、图片链接。
- 不删除图片和代码块。
- 只改排版、段落、标题、列表、表格、空行和轻度语句衔接。
- 只输出最终 Markdown，不加解释。

## 单文件提示词模板

适用于中小文件。

```text
你现在要做的是“技术 Markdown 笔记清洗”，不是改写知识内容。

目标：
把下面这份 Markdown 笔记整理成更适合人类阅读的版本，让它更像一份正常的 Markdown 学习笔记。

严格约束：
1. 不改变知识含义。
2. 不删除有效内容。
3. 不新增原文中没有的知识点。
4. 不修改代码逻辑、命令、接口名、术语、链接、图片链接。
5. 不删除图片和代码块。
6. 只优化排版、段落、标题、列表、表格、空行、语句衔接。
7. 遇到表达跳跃的地方，只做轻度整理，不要重写成另一篇文章。
8. 输出必须仍然是 Markdown。
9. 只输出整理后的 Markdown，不要解释。

下面是原文 Markdown：

{{FILE_CONTENT}}
```

## 超长文件分块模板

适用于超长文件。分块时按自然边界切，不要切在代码块中间。

```text
你现在要整理一份超长技术 Markdown 笔记的其中一部分。

任务目标：
只清理这一段 Markdown 的阅读体验，让它更适合人类阅读。

严格约束：
1. 不改变知识含义。
2. 不删除有效内容。
3. 不新增原文没有的知识点。
4. 不修改代码逻辑、命令、术语、图片链接、普通链接。
5. 不删除图片和代码块。
6. 只做排版、列表、标题、段落、表格层面的整理。
7. 输出仍然必须是 Markdown。
8. 只输出整理后的 Markdown 片段，不要解释。
9. 必须保持这一段的开头和结尾语义连续。

下面是该文件的一个分块：

{{CHUNK_CONTENT}}
```

## 标准工作流

1. 从 `notes/cleaned/` 选一个文件，不处理 `notes/source/`。
2. 根据文件长度选择“单文件模板”或“分块模板”。
3. 只接收 Markdown 输出。
4. 回看差异，重点检查代码块、命令、图片链接、标题层级、表格信息是否被改坏。
5. 确认无误后，再覆盖 `notes/cleaned/` 中对应文件。
6. 完成一个明确批次后，按 [git_workflow.md](git_workflow.md) 提交。

## 原则

如果目标是“可重复、低风险、零语义漂移”，脚本是核心。

如果目标是“更适合复习和长期阅读”，LLM 是补充。

推荐长期组合：

`脚本做确定性清洗 + LLM 做逐文件阅读优化`
