## 3. Linux / WSL 开发环境：先会用，再理解边界

你以前的笔记里提到过 `WSL1`、中文、默认 root、`sudo` 免密、`sshd`、`openssl`、系统日志。这些内容本质上都属于“开发环境搭建”。

### 3.1 WSL 的定位

`WSL` 更适合本地开发、实验和日常命令行使用，不等于真实生产环境。

要注意：

- 生产机器一般是原生 Linux，不是 WSL。
- WSL1 和 WSL2 的网络、文件系统、内核能力差异很大。
- 你在 WSL 上能跑通，不代表部署到真实服务器、容器、CI 环境就完全一致。

对后端工程师来说，正确态度是：

- 把 WSL 当成本地 Linux 练习场。
- 关键知识仍然按真实 Linux 理解，例如文件描述符、进程、信号、socket、日志、权限、动态库。
- 遇到和内核、容器、网络命名空间强相关的问题，要优先以真实 Linux 行为为准。

### 3.2 为什么要看系统日志

你原笔记里提到了 `/var/log/syslog`。这很重要，因为很多初学者只会看自己程序的日志，不会看系统日志。

系统日志常用于排查：

- 服务是否启动失败。
- 端口是否绑定失败。
- 权限是否不足。
- 动态库是否缺失。
- 磁盘、网络、系统服务是否异常。

在现代 Linux 中，你至少要知道两套视角：

- 传统日志文件，例如 `/var/log/syslog`、`/var/log/messages`。
- `systemd` 体系下的 `journalctl`。

对你现在的求职目标来说，不需要背一堆命令，但要能回答：

- 程序起不来时，除了程序自身日志，还看什么。
- 进程退出码、core dump、系统日志分别说明什么。
- 为什么线上排障不能只靠 `printf`。

### 3.3 `sudo` 免密、默认 root 这些设置为什么不能乱用

这些配置在个人开发机上为了方便可以理解，但必须知道风险：

- 默认 `root` 会弱化权限边界意识。
- `sudo` 免密会降低误操作成本。
- 一旦在错误目录执行危险命令，后果比普通用户更大。

工程上更推荐：

- 日常开发用普通用户。
- 必要时再使用 `sudo`。
- 把“权限不足”当作一种正常信号，而不是默认绕过。

这也是面试里常见的隐含考点：你有没有真正的 Linux 使用习惯，而不是只在 IDE 里点按钮。

### 3.4 原笔记中的 `openssl`、`sshd` 应该怎么理解

- `openssl`：不是只会生成证书，而是要知道它和 TLS、HTTPS、证书链、私钥、公钥、握手有关。
- `sshd`：不是只会远程登录，而是和运维接入、部署方式、跳板机、免密登录、远程调试都有关系。

以后学 `HTTP/Nginx`、`Docker`、`分布式系统` 时，这两个点都会再次出现。

## 4. 命令行不是死记硬背，而是理解“对象、作用域、风险”

原文里保留了 Windows 的 `del /s *.ev4` 示例。你不用记住这个命令本身，但要学会一个更通用的工程习惯：

- 任何删除命令先确认当前目录。
- 任何带通配符的命令先确认匹配范围。
- 任何批量操作先做“只列出、不执行”的预检查。

这类习惯在 Linux 下同样重要，例如：

- `rm -rf`
- `find ... -delete`
- `xargs rm`
- 批量替换脚本

面试中如果问你“线上最怕什么误操作”，比起背命令，更好的回答是：

`高权限 + 模糊匹配 + 批量执行 + 没有预检查`

## 5. 服务端经典模型：Reactor + Thread Pool 还重要吗

原笔记里写到：

`Reactor + threadPool 解决 I/O 密集型 + 计算密集型`

这句话方向没错，但太短。真正面试或工作中，要能展开成下面这套说法。

### 5.1 为什么会出现 Reactor

如果一个服务器要同时处理大量连接，不能每个连接都阻塞式等待，否则线程数量、上下文切换和内存开销都会失控。

所以会引入：

- `IO multiplexing`：`select / poll / epoll`
- 事件驱动：有事件才处理
- 非阻塞 socket

Reactor 模式的核心是：

- 一个或多个事件循环负责监听“连接、可读、可写、关闭、错误”等事件。
- 当事件发生时，把它分发给对应的处理器。

### 5.2 为什么还要线程池

事件循环本身擅长处理：

- 接收连接
- 收发数据
- 状态机推进
- 轻量级协议解析

但如果某个请求涉及：

- JSON/XML 大量解析
- 复杂业务计算
- 图片/音视频处理
- 数据压缩解压
- 阻塞式数据库访问

那就不应该卡住事件循环线程，而是要把耗时任务派发给工作线程池。

所以更准确的说法是：

- `Reactor` 负责高并发 I/O 事件分发。
- `Thread Pool` 负责把可能阻塞或耗 CPU 的任务隔离出去。

### 5.3 这个模型今天还重要吗

重要，但不要用过时的方式理解。

云计算、容器、Kubernetes 并没有让这个模型失效，它们解决的是：

- 部署
- 伸缩
- 资源隔离
- 运维编排

而 `Reactor + Thread Pool` 解决的是进程内部的并发组织问题。两者根本不在一个层面。

真正过时的不是这个思想，而是：

- 把所有逻辑都塞进一个巨大的事件回调里。
- 没有背压控制。
- 没有任务队列长度治理。
- 没有超时和连接生命周期管理。

### 5.4 线上常见问题

这类服务经常出的问题包括：

- 某个业务逻辑阻塞了事件循环，导致整个服务 RT 飙升。
- 线程池排队过长，任务堆积，内存持续增长。
- 数据库或下游服务慢，反压传导回接入层。
- 写事件处理不当，发送缓冲区膨胀。
- 连接闲置超时策略缺失，FD 数持续升高。

面试里如果只说“Reactor 很高效”，层次还不够。要能继续说出：

- 为什么要非阻塞。
- 为什么不能在 IO 线程里做重活。
- 怎样识别线程池过载。
- 过载时是扩容、限流、降级还是拒绝。

## 6. 工程代码常用三层：还可以再拆清楚一点

原笔记里的三层是：

- 业务层
- 协议层
- TCP 通道层

这个思路是对的，但工程上建议你以后统一理解为四层：

### 6.1 通道层（Transport / Channel）

关注点：

- socket 生命周期
- 连接建立与关闭
- 收发缓冲区
- 非阻塞 IO
- `epoll` 事件注册与处理
- 心跳、超时、连接状态

典型问题：

- 半包 / 粘包
- 写半包
- 对端关闭
- `EAGAIN`
- FD 泄漏

### 6.2 协议层（Protocol / Codec）

关注点：

- 报文边界
- 编解码
- 序列化与反序列化
- 包头设计
- 版本兼容
- 错误码规范

典型问题：

- 长度字段不可信怎么办
- 协议升级如何兼容旧客户端
- 文本协议与二进制协议如何选

### 6.3 业务层（Service / Domain）

关注点：

- 请求路由
- 参数校验
- 业务编排
- 缓存、数据库、RPC 调用
- 幂等、限流、降级

典型问题：

- 一个接口超时，到底是业务问题还是下游问题
- 如何避免把 SQL 和 socket 代码直接写进 handler

### 6.4 基础设施层（Infrastructure）

很多初学者漏掉这一层，但工作中它很关键：

- 日志
- 配置
- 线程池
- 连接池
- 指标监控
- 链路追踪
- 告警
- 部署脚本

如果只会前三层，你写出来的通常还是 demo；把基础设施补上，才更像能上线的服务。

## 7. 从经典系统书的角度，应该抓哪些“底层骨架”

你后面复习很多内容时，都可以借 `TLPI`、`UNP`、`APUE`、`CSAPP` 这类经典资料的视角统一理解。建议重点抓这几根主线：

### 7.1 一切从“进程 + 文件描述符”开始

在 Unix 世界里，很多对象都能用“文件描述符”这个统一抽象来操作：

- 普通文件
- 管道
- socket
- 某些设备

这能帮你把看似零散的系统调用串起来：

- `open`
- `read`
- `write`
- `close`
- `dup`
- `fcntl`

### 7.2 理解“用户态 / 内核态边界”

很多性能和排障问题，本质都与边界切换有关：

- 系统调用为什么比普通函数重
- 为什么频繁小包发送会低效
- 为什么上下文切换、锁竞争、缓存失效会影响性能

### 7.3 进程模型是后面网络与服务治理的基础

要真正理解服务器，不只是会 `socket`，还要理解：

- 父子进程关系
- 僵尸进程
- 信号
- 守护进程
- 会话与进程组

这些知识在“服务启动、重启、退出、拉起、日志重定向、优雅关闭”里都能用到。

### 7.4 读 APUE 不是为了背接口，而是为了建立系统观

真正该形成的意识是：

- 一个服务启动时做了哪些事。
- 一个连接进入进程后经历了哪些层次。
- 一个进程异常退出后，系统层面会留下什么痕迹。

## 8. 你在工作中很可能遇到的典型问题

下面这些问题，比“背出三次握手四次挥手”更接近真实工作。

### 8.1 网络与协议

- TCP 是字节流，应用层必须自己设计消息边界。
- 一次 `send` 成功不等于对方业务层已经处理成功。
- 一次 `read` 读到 0，通常意味着对端关闭。
- 非阻塞 IO 下，`EAGAIN` 不是错误，而是“现在不能继续”。

### 8.2 并发与线程池

- 把数据库查询直接放在 IO 线程里，会拖慢整个事件循环。
- 线程池不是越大越好，线程过多会导致切换开销和锁竞争加重。
- 队列无上限时，系统常见死法不是 CPU 打满，而是内存先涨死。

### 8.3 MySQL 与下游依赖

- 服务慢不一定是代码逻辑问题，也可能是慢 SQL、锁等待、连接池耗尽。
- 一个接口超时，要拆成接入耗时、排队耗时、业务耗时、DB 耗时、下游 RPC 耗时分别看。

### 8.4 排障与可观测性

- 没有结构化日志，线上就只能“猜”。
- 没有请求 ID，跨模块排查会非常痛苦。
- 没有基础指标，你根本不知道是 QPS 上去了、RT 变长了，还是错误率上来了。

这部分内容，后面要不断和 `MySQL`、`网络编程`、`分布式系统` 几篇文档联动起来。

### 10.1 主线一：Linux C++ 服务端工程化

### 10.2 主线二：偏设备 / 机器人 / 边缘方向的 C/C++

这是你长期抗替代性更强的一条观察线，但不建议立刻把主战场切过去，因为你目前硬件、RTOS、驱动、电控基础还薄。

更稳妥的策略是：

- 保持主线仍是 Linux C++。
- 同时补嵌入式基础：`STM32 + C + 中断 + 定时器 + UART/SPI/I2C + FreeRTOS`
- 再决定是否往“机器人嵌入式软件 / 工业控制 / 边缘设备”深入。

这条线里，`MQTT` 值得学，但它只是协议，不是全部。真正要补的是：

- 串口与总线通信
- 实时性与任务调度
- 状态机
- 设备侧调试方法
- 通信可靠性

### 10.3 主线三：AI 协作式工程能力

你不需要转成“纯 AI 应用开发”，但一定要把 AI 用成工程放大器。

你应该让 AI 帮你做：

- 架构拆解
- 接口定义
- 时序图和模块图草稿
- 代码审查清单
- 测试点枚举
- 故障树分析
- 简历和项目表达优化

不要让 AI 代替你思考“系统边界、线程模型、故障处理、性能瓶颈”。这些正是以后区分初级和高级工程师的地方。

- 如果补齐日志、监控、限流、超时、线程池、连接池，成长会很扎实。

风险：

- 岗位绝对数量不如 Java 多。
- 很多面试会追问项目细节、性能、排障，而不只是语法八股。
- 如果只会接口 CRUD，会被压在偏低端的岗位区间。

结论：

- 这是你近期求职的主战场。

### 11.2 Linux 嵌入式应用开发

这是你很值得观察的第二主线，而且比“直接切纯 MCU 底层驱动”更适合你。

这里说的不是纯裸机，而是：

- ARM Linux 设备侧应用
- 设备网关
- 边缘计算节点
- 工业设备通信服务
- 机器人上位机 / 调度 / 任务编排 / 通信模块

优点：

- 仍然大量使用 C/C++，和 Linux 服务端有很强的知识复用。
- 更接近真实设备、协议、现场问题，长期抗替代性更好。
- 通常比纯互联网平台开发更看重稳定性和问题定位能力。

风险：

- 需要补交叉编译、设备调试、串口/总线通信、部署环境限制。
- 有些岗位名字叫“嵌入式”，实际要求很杂，要分清是 `Linux 应用` 还是 `驱动 / BSP / RTOS`。

结论：

- 这是你最值得并行培养的备选方向。

### 11.3 机器人嵌入式软件 / 控制软件

这个方向的天花板高，长期也不容易被简单替代，但你暂时不要直接冲“最底层控制”。

你当前更匹配的切入点是：

- 机器人通信与设备接入
- 上位机控制软件
- 调度与任务编排
- 视觉与机械臂之间的中间层
- 边缘控制网关

不建议立刻主攻的点：

- 电机控制算法
- 高速伺服闭环
- FPGA
- 复杂动力学与实时控制器实现

原因不是你学不会，而是你当前短板不在“会不会一点 ROS”，而在：

- 硬件调试经验不够
- 控制链路工程经验不够
- 实时系统理解不够
- 数学到代码的映射还不稳

结论：

- 可以长期观察，但第一份转岗不建议直接打最硬核的控制底层。

### 11.4 新能源 / 工业控制方向

这个方向能进，不是双非硕士就完全没机会，但要分清岗位类型。

更适合你观察的岗位：

- 充电桩 / 能源网关 / EMS 边缘软件
- 设备接入平台
- 工业通信服务
- 数据采集与监控后台

不适合立刻硬切的岗位：

- 电池管理底层算法
- 电机控制
- 功率电子强相关控制
- 强硬件协同验证岗位

行业判断：

- 长期需求是真实存在的。
- 但加班强度和企业文化差异很大，制造业节奏、交付周期、出差比例都要单独判断。
- 如果目标是“稳定长期耕耘”，要尽量选软件边界清晰、不是全年跟产线救火的团队。

### 11.5 全屋智能 / IoT

全屋智能不是没有空间，但切入点不在“再做一个米家”。

更现实的机会在：

- 多协议设备接入
- 本地网关
- 边缘自动化
- B 端楼宇 / 园区 / 工业场景
- 与安防、能耗、门禁、传感器联动的系统

要注意：

- 消费级智能家居 APP 层竞争很卷，平台生态被大厂占得很深。
- 真正有技术壁垒的，往往是设备互联、稳定性、离线能力、本地控制、协议兼容和交付能力。

### 11.6 AI 出海独立开发

这条线可以保留为副线，但不建议当成当前求职主线。

原因很简单：

- 你现在的核心资产在 C++、Linux、系统能力，不在流量、营销、选品和增长。
- `vibe coding` 能加快做站，但不能替你解决真实需求、获客和转化。
- 它更适合当副项目，用来练产品感和 AI 协作，不适合替代当前主业求职。

### 11.7 对你的推荐结论

按你现在的背景，我建议优先级这样排：

1. `Linux C++ 服务端 / 网络通信 / 网关 / 设备接入平台`
2. `Linux 嵌入式应用 / 边缘设备 / 机器人上位机或通信层`
3. `工业软件 / 新能源边缘软件 / IoT 网关`
4. `更底层的 MCU / RTOS / 驱动 / 实时控制`
5. `AI 独立开发`

这意味着你现在最优策略不是“彻底转方向”，而是：

- 求职主线仍然围绕 `Linux C++`
- 把岗位关键词往 `设备侧 / 工业 / 边缘 / 通信` 扩展
- 用 STM32 和协议学习，逐步建立转向嵌入式应用的能力

## 12. 不要过早追“架构师”头衔，先变成能扛模块的人

你以后当然可以往架构方向走，但不要把“架构师”理解成只会画框图的人。

真正有价值的高级工程师 / 架构型工程师，必须同时懂：

- 模块边界
- 并发模型
- 数据流
- 资源模型
- 故障模型
- 部署与观测

一个比较现实的成长顺序是：

1. 能独立写完功能。
2. 能独立负责一个模块。
3. 能把模块做成可维护、可观测、可排障。
4. 能拆一个子系统，并说清楚为什么这么拆。
5. 能在业务、性能、成本、复杂度之间做取舍。

所以你当前最该追求的不是“我要不要转架构”，而是：

- 我能不能把一个服务讲成清晰的层次。
- 我能不能把一次故障定位过程讲得专业。
- 我能不能把系统的瓶颈、超时、资源泄漏、扩展性说清楚。

等你能稳定做到这些，“架构能力”自然会长出来。

## 13. 如果往嵌入式应用走，基础应该怎么补

你已经买了 `STM32F103C8T6`，这是合适的起点，但不要一上来就急着碰复杂 RTOS 项目。

建议补法按这个顺序走：

### 13.1 先补硬件最小闭环

- GPIO
- 中断
- 定时器
- UART
- SPI
- I2C
- ADC / PWM 的基本概念

同时建议准备的工具优先级是：

1. 万用表
2. USB 转串口
3. 逻辑分析仪
4. 示波器

对初学者来说，万用表和串口调试工具比示波器更快产生反馈。

### 13.2 再补任务调度和状态机

- 裸机轮询和中断驱动有什么区别
- 什么时候需要状态机
- 什么时候引入 `FreeRTOS`
- 定时任务、消息队列、互斥和优先级反转是什么意思

### 13.3 通信协议按“由近到远”补

如果你以后想做设备侧或 IoT，不需要一开始把所有协议都学全，但至少要形成一条梯度：

- 板级与芯片间：`UART / SPI / I2C`
- 设备总线：`CAN`
- 设备接入：`Modbus RTU / Modbus TCP`
- 设备到云：`MQTT`
- 工业实时控制再往后看：`CANopen / EtherCAT`

对你而言，`MQTT` 值得学，而且很实用，但它更偏“设备到平台”的应用层协议。它不能替代你对串口、总线、状态机和可靠传输的理解。
