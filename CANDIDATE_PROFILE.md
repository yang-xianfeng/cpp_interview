# Candidate Profile

这份文档是统一背景设定。后续做学习规划、项目包装、简历润色、面试准备时，默认以这里为准。

原则：

- 只保留会影响职业判断、项目包装和面试表达的背景。
- 不追求把所有零散技术点都堆成清单。
- 在完整和简洁之间，优先保留高价值信息。

## 当前定位

- 双非硕士，跨考 `408` 计算机方向。
- 当前身份是 `初级 Linux C++ 后端 / 服务器开发工程师`。
- 主要经历偏 `API 逻辑`、`C/S demo`、基础服务端开发。
- 当前目标不是继续堆 demo，而是成长为 `能负责模块、能讲清系统、能逐步走向架构` 的工程师。

## 技术背景

- `C / C++`：具备常规开发基础，理解类、对象、模板、STL 等常见内容，但工程化深度仍需补强。
- `Linux`：熟悉常用环境、命令行、进程管理、基础排障，对 Unix 风格开发有基本认知。
- `服务端`：接触过 `socket`、`epoll`、进程管理与通信、线程、MySQL，做过 `C/S demo` 和基础接口逻辑开发。
- `前后端配合`：做过 `HTML + CSS + JS` 或前端框架的简单配合开发。
- `机器人`：接触过 `Fanuc` 工业机器人，独立完成过 `FANUC CR-35iA` 模型导入、`MoveIt` 控制和部分 `ROS C++` 驱动。
- `视觉 / 点云`：对机器视觉、`3D` 点云处理有概念性理解，了解 `ICP` 等基础内容。
- `嵌入式`：几乎没有真正的硬件和嵌入式项目经验，已购入 `STM32F103C8T6` 套件，准备从基础补起。
- `AI 工具`：用过 AI 对话、`Cursor`、`Gemini CLI`，知道 `role / rules / agent / skills` 这类协作约定；对 `Claude Code`、`Codex`、`Hermes`、`harness` 一类新工具链有关注，但还没有形成稳定工作流。

## 可作为卖点的部分

- 有 `Linux C++ 服务端` 的基础链路认知，能把 `socket + epoll + MySQL + 进程线程` 讲成一条主线。
- 有基础的服务端实现经验，知道 API 逻辑、简单前后端配合和 C/S 程序开发。
- 有一定机器人软件探索经历，不是纯课堂型背景。
- 对设备侧、视觉侧、服务端之间的交叉方向有真实兴趣和早期积累。

## 当前短板

- 缺少真正可上线、可观测、可排障的服务经验。
- 项目表达仍偏 demo 化，对模块边界、线程模型、性能治理、故障处理讲得不够成熟。
- 嵌入式、硬件、电控、RTOS、驱动基础薄弱。
- 数学到代码的映射能力偏弱，尤其是李代数、机器人动力学实现。

## 求职偏好

- 排序：`可持续 > 薪资稍好 > 不卷`
- 不追求长期高强度加班。
- 希望做能积累 10 到 20 年、且不容易被低门槛替代的方向。
- 愿意拥抱 AI，但不希望转向纯流量或纯概念路线。

## 学习主线

- 主线一：`Linux C++ 服务端工程化`
  重点补 `网络模型、线程池、连接池、MySQL、缓存、配置、日志、监控、性能与排障`
- 主线二：`设备侧 / 嵌入式应用 C/C++`
  重点补 `STM32、中断、定时器、串口、SPI / I2C / CAN、状态机、FreeRTOS、MQTT / Modbus`
- 主线三：`AI 协作式工程能力`
  重点补 `架构拆解、测试点设计、排障清单、文档和项目表达优化`

## 项目包装边界

- 只在真实经历基础上做工程化提升和表达重构，不凭空虚构。
- 包装重点是：模块职责、技术选型、边界、故障、优化、排障。
- 目标是沉淀两段真实、完整、可追问的项目经历。

## AI 协作原则

AI 主要负责：

- 梳理学习路径
- 解释底层原理
- 拆系统和模块
- 补常见故障与检查清单
- 优化项目、简历、面试表达
- 在复习阶段补充必要知识点、经典资料结论、源码切面和面试官式追问

AI 不替代：

- 技术边界判断
- 项目真实性把关
- 关键取舍和责任判断
